Sandbox Engine v0.4.2 – Moteur de démonstration GHI

1. Rôle du moteur GHI

Le Sandbox Engine est un moteur de calcul synthétique utilisé pour démontrer la méthodologie du Global HashCost Index (GHI) et pour tester l’API publique dans un environnement stable.

Il ne consomme pas de données de production : toutes les valeurs sont générées de manière déterministe afin d’offrir un cadre de test parfaitement reproductible pour les équipes techniques et les institutions intéressées par le standard GHI.

Important : le Sandbox Engine ne doit pas être utilisé tel quel pour prendre des décisions d’investissement. Il s’agit d’un moteur de démonstration, conçu pour l’évaluation méthodologique et l’intégration technique.

2. Version courante : Sandbox Engine v0.4.2

La version actuellement déployée sur le site public est :

  • Engine : v0.4.2-sandbox
  • API principale : API v1.1 (compatible v1.0)
  • Méthodologie : v1.0.0

Principales caractéristiques de la v0.4.2

  • Alignement complet avec la spécification API v1.1 : réponses unifiées avec version, timestamp_utc, engine_sandbox_version.
  • Structure normalisée des régions utilisée par /v1/ghi/regions et /v1/ghi/snapshot.
  • Ajout des endpoints /metadata, /network et /forecast pour les démonstrations de cas d’usage institutionnels.
  • Données de coûts de production cohérentes entre snapshot, historique et statistiques globales.
  • Tous les tests automatisés du dépôt ghi-engine passent avec succès (suite de tests CI dédiée au Sandbox Engine).

3. Architecture fonctionnelle (vue synthétique)

Le moteur sandbox est volontairement compact et transparent. Il se compose de plusieurs blocs principaux :

  • Générateur de réseau : produit des valeurs synthétiques de hashrate, difficulté et frais moyens, cohérentes avec un scénario de marché.
  • Module de coûts régionaux : génère, pour chaque région GHI, des coûts min / moyen / max en USD par BTC.
  • Calculateur de statistiques globales : agrégation des coûts régionaux en statistiques globales (min / avg / max).
  • Adaptateur API : formatage des réponses pour l’API publique (/v1/ghi/snapshot, /v1/ghi/history, etc.).
  • Bloc de métadonnées : insertion systématique de engine_sandbox_version et des métadonnées de version dans les réponses.

Cette architecture est conçue pour être lecture facile côté code (Python/FastAPI) et pour servir de base pédagogique avant l’introduction d’un moteur connecté à des données réelles.

4. Endpoints servis par le moteur

Les endpoints suivants de l’API GHI sont actuellement alimentés par le Sandbox Engine v0.4.2 :

Endpoint Description Statut
/v1/ghi/snapshot Coûts de production régionaux (min / avg / max) + statistiques globales. Stable
/v1/ghi/history Séries historiques synthétiques de coûts agrégés. Stable
/v1/ghi/regions Liste des régions GHI exposées par le modèle. Stable
/v1/ghi/metadata Métadonnées de l’API et du moteur (version, sandbox, etc.). Stable
/v1/ghi/network Métriques réseau synthétiques (hashrate, difficulté, frais moyens). Beta
/v1/ghi/forecast Projection synthétique du coût moyen de production. Beta

Pour le détail complet des schémas de réponse, se référer à la documentation API v1.1 et à la spécification technique GitHub.

5. Garanties pour les usages institutionnels

Même si les données sont synthétiques, le Sandbox Engine v0.4.2 applique déjà plusieurs principes clés attendus par les institutions :

  • Déterminisme : mêmes entrées, mêmes sorties (aucune surprise sur les pipelines de test).
  • Versionnement explicite : la version du moteur est exposée dans les réponses API via engine_sandbox_version.
  • Stabilité de schéma : alignement strict sur les schémas API publiés (API v1.1).
  • Transparence du code : moteur open-source sur GitHub, documenté et testé.

Ces garanties permettent aux équipes d’intégration de préparer leurs pipelines, validations de conformité et environnements de test avant l’arrivée de versions connectées à des données réelles.

6. Roadmap moteur (au-delà du v0.4.2)

Les travaux prévus sur les prochaines versions du moteur incluent notamment :

  • intégration progressive de données réelles (coût énergie, mix machines, etc.) ;
  • segmentation plus fine par sous-régions et profils énergétiques ;
  • amélioration des profils temporels (saisonnalité, chocs de marché, halving) ;
  • exposition d’indicateurs avancés pour la recherche académique et institutionnelle.

Toutes les évolutions significatives seront documentées dans le Changelog et dans les dépôts GitHub associés.

Sandbox Engine v0.4.2 – GHI Demonstration Engine

1. Engine purpose

The Sandbox Engine is a synthetic compute engine used to demonstrate the Global HashCost Index (GHI) methodology and to exercise the public API in a stable environment.

It does not ingest production data: all values are deterministically generated so that technical and institutional users can rely on fully reproducible results.

Important: the Sandbox Engine must not be used as is for investment decisions. It is a demonstration engine, designed for methodological review and technical integration.

2. Current version: Sandbox Engine v0.4.2

The public website currently runs:

  • Engine: v0.4.2-sandbox
  • Main API: API v1.1 (v1.0 compatible)
  • Methodology: v1.0.0

Key properties of v0.4.2

  • Full alignment with the API v1.1 specification: unified responses with version, timestamp_utc, engine_sandbox_version.
  • Normalized regional structure shared by /v1/ghi/regions and /v1/ghi/snapshot.
  • New /metadata, /network and /forecast endpoints for institutional-grade demonstrations.
  • Internally consistent production-cost figures across snapshot, history and global statistics.
  • All automated tests in the ghi-engine repository pass successfully, including the dedicated Sandbox Engine test suite.

3. High-level architecture

The sandbox engine is intentionally compact and transparent. It is built from the following components:

  • Network generator: produces synthetic hashrate, difficulty and average fee values.
  • Regional cost module: generates min / avg / max USD per BTC for each GHI region.
  • Global statistics calculator: aggregates regional costs into global min / avg / max statistics.
  • API adapter: shapes responses for the public API (/v1/ghi/snapshot, /v1/ghi/history, etc.).
  • Metadata block: consistently injects engine_sandbox_version and version metadata into responses.

The code is designed to be easy to read (Python/FastAPI) and to serve as a pedagogical step before introducing an engine backed by live data.

4. Endpoints powered by the engine

The following API endpoints are currently powered by the Sandbox Engine v0.4.2:

Endpoint Description Status
/v1/ghi/snapshot Regional production costs (min / avg / max) + global statistics. Stable
/v1/ghi/history Synthetic aggregated time series of production costs. Stable
/v1/ghi/regions List of GHI regions exposed by the model. Stable
/v1/ghi/metadata API and engine metadata (version, sandbox status, etc.). Stable
/v1/ghi/network Synthetic network metrics (hashrate, difficulty, average fees). Beta
/v1/ghi/forecast Synthetic forecast of average production cost. Beta

For full response schemas, see the API v1.1 documentation and the technical specification on GitHub .

5. Institutional-grade guarantees

Even though the data is synthetic, Sandbox Engine v0.4.2 already applies several principles expected by institutional users:

  • Determinism: same inputs, same outputs (no surprises in automated pipelines).
  • Explicit versioning: the engine version is exposed via engine_sandbox_version in API responses.
  • Schema stability: strict alignment with the published API v1.1 schemas.
  • Code transparency: open-source engine on GitHub, documented and fully tested.

This allows integration teams to prepare ETL pipelines, compliance checks and test environments ahead of future, data-backed engine versions.

6. Engine roadmap (beyond v0.4.2)

Planned evolutions include:

  • progressive integration of real-world inputs (energy costs, ASIC mix, etc.);
  • finer regional segmentation and energy profiles;
  • richer temporal patterns (seasonality, market shocks, halvings);
  • advanced indicators for academic and institutional research.

All major changes will be documented in the Changelog and in the relevant GitHub repositories.
