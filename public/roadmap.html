Roadmap du Global HashCost Index

Le Global HashCost Index (GHI) suit une stratégie de déploiement en plusieurs phases : d’abord la publication du standard, puis une API sandbox structurée, et enfin le moteur de production avec données réelles et SLA institutionnels.

Phase 1 – Standard & Documentation (complétée)

Statut : terminée

Objectif : publier un socle public, auditable et lisible pour les institutions, sans exposer le moteur interne.

Livrables clés

  • Publication du whitepaper GHI (FR & EN).
  • Publication du document Méthodologie & Transparence (FR & EN).
  • Document GHI for Institutions (FR & EN).
  • Standard technique GHI Technical Standard v1.0.
  • Modèle de données GHI Data Model v1.0.
  • Site statique GHI-public avec pages Overview, Méthodologie, Régions, Gouvernance, Roadmap, API.

But institutionnel

  • Figé l’antériorité du standard GHI.
  • Fournir une base compréhensible pour régulateurs, risk managers, analysts.
  • Permettre des premières revues méthodologiques sans exposer les données internes.

Phase 2 – Moteur Sandbox & API (en cours)

Statut : en cours

Objectif : proposer une API fonctionnelle mais alimentée en fake data structurées, parfaitement alignées sur le Data Model v1.0. Cette étape permet de tester les intégrations sans risque opérationnel.

Composants techniques

  • Référentiel ghi-engine dédié au moteur et à l’API.
  • API FastAPI (sandbox) exposant :
    • GET /v1/ghi/snapshot
    • GET /v1/ghi/history
    • GET /v1/ghi/regions
    • GET /v1/ghi/regions/{region_id}
    • GET /v1/ghi/stats
  • Données synthétiques conformes au GHI Data Model v1.0 (snapshot, historique, régions, stats).
  • Tests automatisés de non-régression (Pytest) sur les endpoints principaux.

Objectifs de la phase

  • Valider le design de l’API avec des équipes techniques externes.
  • Permettre aux institutions de construire des dashboards/tests sur une interface stable.
  • Préparer l’infrastructure pour le moteur réel (config, sécurité, monitoring).

Étapes restantes sur la Phase 2

  • Documenter la sandbox dans la documentation publique (API_REFERENCE.md et api.html).
  • Clarifier le futur endpoint public (domaine, authentification, limites de débit).
  • Préparer les gabarits de reporting (exemples de JSON/CSV pour les institutions).

Phase 3 – Moteur de Production & API réelle (à venir)

Statut : planifiée

Objectif : activer le moteur complet de calcul du coût de production du Bitcoin, alimenté par des données réelles (hashrate, mix hardware, énergie, régions), avec un niveau de qualité compatible avec une utilisation institutionnelle.

Fonctionnalités prévues

  • Activation du moteur de calcul réel (par région, agrégation globale min/avg/max).
  • Intégration des données de hashrate régionales issues de fournisseurs spécialisés.
  • Mise à jour régulière des paramètres machines (efficacité, générations ASIC, etc.).
  • Snapshots quotidiens (puis intrajournaliers si nécessaire).
  • Historique complet GHI exploitable en API et via exports CSV.

Exigences institutionnelles

  • Versioning strict des modèles (Data Model, Technical Standard, moteur).
  • Processus de gouvernance définis (validation des mises à jour, changelog public).
  • Documentation renforcée sur les hypothèses régionales et les sources de données.
  • Possibilité d’ajouter des extensions : coûts CO₂, fiscalité, granularité sous-régionale.

Extensions envisagées

  • Intégration de métriques carbone (par région, par kWh, par BTC produit).
  • Affinage des régions (ex : Texas, Québec) lorsque les données sont suffisamment robustes.
  • Exports avancés (CSV/Parquet) pour les équipes quant et risk.

Gouvernance et mises à jour

Chaque évolution majeure (nouvelle version de moteur, ajout de régions, changements méthodologiques significatifs) sera publiée avec :

  • un tag de version dans les dépôts publics concernés ;
  • un changelog clair pour les institutions ;
  • une mise à jour de la page Roadmap et des documents standards (Tech Standard, Data Model).
